Segmented Control is a compact single-select: two or more mutually exclusive segments in a pill-shaped track, with exactly one active at a time. The Blueprint instance ships with two fixed segments, "Buy" and "Rent". It's the control for switching between a small set of related views or filters where all options should stay visible.

## When to use it

Use a segmented control to switch between a small number (two, ideally not more than four or five) of mutually exclusive views, modes, or filters that the user benefits from seeing all at once. "Buy / Rent" is the canonical case: two persistent, equally weighted choices that frame everything below them.

Reach for a different control when:

- **The options are a form input you submit** — use radio buttons or a dropdown. A segmented control reads as an immediate view switch, not a field you fill in.
- **There are more than ~5 options, or the labels are long** — segments get cramped and the labels truncate. Use a dropdown.
- **The choice triggers an action rather than switching a view** — use buttons.
- **It's a binary on/off setting** — use a toggle.

The distinction from a dropdown matters: a segmented control keeps every option visible and switches instantly, which suits primary, high-frequency choices. A dropdown hides options behind a click, which suits longer or lower-frequency lists.

## Labels

Keep segment labels to one or two words so they fit the fixed track without truncating. Use parallel phrasing across segments ("Buy / Rent", not "Buy / For rent"). The control is single-select and always has exactly one active segment, so there's no "none selected" state to design for.

## Accessibility essentials

The detail lives in the developer spec below; these are the points a designer needs to get right.

<AccordionGroup>
  <Accordion title="It's a radio group, not a row of buttons" icon="square-check">
    Screen readers announce it as a radio group with each segment a radio option ("Buy, selected, 1 of 2"). That selection semantic is the whole point — don't let it be built as plain buttons, which lose the one-of-many relationship.
  </Accordion>

  <Accordion title="One tab stop, arrow keys move selection" icon="arrows-left-right">
    The group is a single tab stop. Arrow keys move between segments (roving focus). Tab moves past the whole control, not through each segment.
  </Accordion>

  <Accordion title="The focus ring is decorative" icon="eye-slash">
    The offset outline that appears on focus is a visual ring only. It must never be announced.
  </Accordion>

  <Accordion title="Disabled removes the whole control from focus order" icon="ban">
    When disabled, neither segment is reachable. The current selection stays visible but the control is inert — give context for why if it matters.
  </Accordion>
</AccordionGroup>

  <Tab title="API">
    > Single-select with two fixed segments, "Buy" and "Rent". Selection is mutually exclusive, controlled by `value`. The Figma `State` axis encodes interaction states; Hover and Focus are transient and not API props, while Disabled is exposed as `isDisabled`. Segment labels are fixed in this design instance (the canonical `segments` array isn't consumer-editable here).

    ### Properties

    | Property | Type | Values | Default | Notes |
    | --- | --- | --- | --- | --- |
    | segments | array | Segment\[\] | – | Ordered array of selectable segments. This instance is designed with two fixed segments, "Buy" and "Rent"; each is a text label. |
    | value | enum | Buy \| Rent | Buy | Currently selected segment (single-select). Maps to the Figma "Option" axis. |
    | isDisabled | boolean | true \| false | false | Disables the whole control. Maps to Figma "State=Disabled". |

    <Accordion title="Worked examples">
      - **Buy selected (default)** — `value: "Buy"`, `isDisabled: false`
      - **Rent selected** — `value: "Rent"`
      - **Disabled** — `value: "Buy"`, `isDisabled: true`
    </Accordion>

  <Tab title="Structure">
    > Dimensions are fixed (141 × 32) and byte-identical across the Buy/Rent options and the Default, Hover, and Disabled states. Buy and Rent are kept as separate columns in the source to document that intentional symmetry. The selected segment carries a 0.5 hairline outline (outside align) forming the pill "thumb"; the unselected segment has no border. Focus is the only state that changes structure — it adds an offset outline ring. The two Segment layers are plain decorative frames, not component instances.

  <Tab title="Voice / Screen reader">
    > Expose it as a group with radio selection semantics — `radiogroup` on Web, `RadioGroup` / `selectableGroup` on Android, a container element on iOS — with each segment a radio child. One tab stop; arrow keys move selection (roving focus), so only the selected segment is in the Tab order. Each segment's label merges into its accessible name. Selection is announced via state (`aria-checked` / `.isSelected` / `selected`), with "X of 2" set position. The focus-ring rectangle is decorative. Disabled removes the whole control from focus order.

    ### Focus order

    | # | Part | Announcement | Role |
    | --- | --- | --- | --- |
    | 1 | Group | "Buy or Rent, radio group" | radiogroup (single tab stop) |
    | 2 | Selected segment | "Buy, selected, radio button, 1 of 2" | radio (in Tab order, roving) |
    | 3 | Unselected segment | "Rent, radio button, 2 of 2" | radio (via arrow keys) |

    ### Rest — ARIA

    | Property | Value | Notes |
    | --- | --- | --- |
    | group role | radiogroup | Establishes the single-select set; announced as context. |
    | segment role | radio | Prefer native `<input type="radio">` styled as a segment. |
    | aria-checked | true on selected, false otherwise | Marks the active segment. |
    | tabindex | 0 on selected, -1 on others | Roving tabindex; arrow keys move selection. |
    | aria-posinset / aria-setsize | 1 / 2, 2 / 2 | Set position announced as "X of 2". |
    | Do NOT | announce the focus ring | Decorative; `aria-hidden="true"` or render via CSS `:focus-visible`. |

    ### Disabled — ARIA

    | Property | Value | Notes |
    | --- | --- | --- |
    | aria-disabled | true | Set on the group and each radio; prefer native `disabled` so they leave the Tab order. |
    | Do NOT | keep segments focusable | Use native `disabled` or remove from tab order — not `aria-disabled` alone on a div. |

    <Accordion title="iOS (VoiceOver) & Android (TalkBack)">
      **VoiceOver** — group is a container with `.adjustable` (or `.tabBar`); selected segment `[.isButton, .isSelected]` \+ `accessibilityValue` "1 of 2"; unselected drops `.isSelected`; disabled uses `.notEnabled`. **TalkBack** — `Modifier.selectableGroup()` \+ `isTraversalGroup` on the container; segments `Role.RadioButton` with `selected` / `stateDescription`; disabled via `disabled()` on group and segments. In both, merge each label via the platform mechanism, never announce a segment without its parent group, and keep stop indices stable across states.
    </Accordion>
